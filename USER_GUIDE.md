# DeepLens User Guide

This guide explains how to configure DeepLens for **your** hardware/class of
deployment. DeepLens is built to run everywhere — a basic CPU laptop, a machine
with a GPU, or fully in the cloud — by swapping the backend via a single config
flag (`DEEPLENS_MODE`) and toggling a few search-quality settings.

> All settings are environment variables prefixed with `DEEPLENS_`, defined in
> `.env` (copy from `.env.example`). Search/retrieval settings apply to **both**
> modes unless noted.

---

## 1. CPU-only users (basic / privacy-first)

**Typical setup:** old laptop or a private machine, no GPU, runs fully offline.
Mode: **Local** (`DEEPLENS_MODE=local`). Vector DB = **LanceDB**, embedding =
**Jina-CLIP-v2** (local), chat/rewrite = **Ollama** (Llama-3.2-3B / Phi-4-mini).

**Recommended settings:**

```env
DEEPLENS_MODE=local
DEEPLENS_SEARCH_MODE=hybrid
DEEPLENS_ENABLE_DOCUMENT_SUMMARIES=true
DEEPLENS_ENABLE_RERANKER=false      # keep OFF — zero cost, no model download
```

**Why these:** The `hybrid` mode already combines vector + corpus-wide lexical +
summary signals via Reciprocal Rank Fusion — that fusion *is* your re-ranker, and
it costs nothing on CPU. Leave the cross-encoder re-ranker **off**; enabling it
would download a model and add inference even though it is capped to 15 pairs.

**Worth knowing (CPU):**
- Summaries are generated by your local Ollama model during indexing. If Ollama
  is not running, summaries are skipped and `summary` / `find_documents` queries
  will return nothing for that index — re-index with Ollama up.
- **LanceDB lexical fallback:** if the LanceDB full-text index cannot be built,
  keyword search falls back to loading the *entire filtered table* into memory per
  query. Fine for thousands of files; for very large local collections it can be
  slow/RAM-heavy. (The pgvector/GIN path avoids this — see Cloud section.)
- First indexing of many documents is CPU-bound (embedding + optional Whisper/OCR).
  Expect it to take time; it runs in the background.

## 2. GPU users (local, accelerated)

**Typical setup:** a workstation/laptop with an NVIDIA (or Apple-Silicon) GPU.
Still Mode **Local**, but you can afford a heavier re-ranker and larger batches.

**Recommended settings:**

```env
DEEPLENS_MODE=local
DEEPLENS_SEARCH_MODE=hybrid
DEEPLENS_ENABLE_DOCUMENT_SUMMARIES=true
DEEPLENS_ENABLE_RERANKER=true
DEEPLENS_RERANK_TOP_N=25          # raise vs CPU's 15 for more recall before rerank
DEEPLENS_RERANKER_MODEL=cross-encoder/ms-marco-MiniLM-L-6-v2
```

Install the re-ranker dependency first:

```bash
pip install -e ".[reranker]"
```

**Why these:** With a GPU the cross-encoder re-ranker is cheap, so turning it on
with a slightly larger `rerank_top_n` improves final ranking quality. The model
is tiny (~80 MB) and is loaded once per process (cached).

**Worth knowing (GPU):**
- `rerank_top_n` bounds latency and memory: each fused candidate is one
  (query, passage) cross-encoder inference. 25 is comfortable on a GPU; only push
  higher if you have headroom and want maximum recall.
- If you don't want the extra dependency, stay at the CPU profile — quality is
  still strong thanks to RRF fusion.

## 3. Cloud users (enterprise / showcase)

**Typical setup:** Mode **Cloud** (`DEEPLENS_MODE=cloud`). Vector DB =
**PostgreSQL + pgvector**, embedding = **Gemini Text Embedding 004**, chat =
**Gemini 2.5 Flash**. API key stored in the OS keyring (never plaintext).

**Recommended settings:**

```env
DEEPLENS_MODE=cloud
DEEPLENS_SEARCH_MODE=hybrid
DEEPLENS_ENABLE_DOCUMENT_SUMMARIES=true
DEEPLENS_ENABLE_RERANKER=true
DEEPLENS_RERANK_TOP_N=50          # cloud has spare compute — maximize recall
DEEPLENS_HYBRID_RRF_K=60
```

**Why these:** In cloud mode keyword search uses a **real `tsvector` + GIN index**
(Postgres full-text search) — it ranks *every* document efficiently, with no
in-memory table scan. You can therefore afford the highest `rerank_top_n` for the
best final ranking.

**Worth knowing (Cloud):**
- The pgvector lexical path does **not** have the LanceDB in-memory fallback
  limitation — it is the recommended backend for very large corpora.
- Gemini embeddings are 3072-dim; the local Jina-CLIP path is 1024-dim. The schema
  and indexes are created with the correct dimension automatically per mode.
- Re-indexing an existing LanceDB index to add summaries is not needed in cloud
  mode: pgvector migrates the new columns automatically on `initialize()`; just
  re-index the folder to populate summaries.

## Cross-cutting settings reference

| Setting | Env var | Default | Notes |
|---------|---------|---------|-------|
| Search strategy | `DEEPLENS_SEARCH_MODE` | `hybrid` | `chunk` / `summary` / `hybrid` |
| RRF constant | `DEEPLENS_HYBRID_RRF_K` | `60` | Higher = individual ranks matter less |
| Re-ranker on | `DEEPLENS_ENABLE_RERANKER` | `false` | Off by default (CPU-safe) |
| Re-ranker model | `DEEPLENS_RERANKER_MODEL` | `cross-encoder/ms-marco-MiniLM-L-6-v2` | Tiny, CPU-friendly |
| Re-rank pool | `DEEPLENS_RERANK_TOP_N` | `15` | Candidates scored by re-ranker before top_k |
| Summaries on | `DEEPLENS_ENABLE_DOCUMENT_SUMMARIES` | `true` | Generate per-file summary records |
| Summary max chars | `DEEPLENS_SUMMARY_MAX_CHARS` | `4000` | Text fed to the summarizer |

## Quick "which mode should I use?" cheat-sheet

- Looking for a **specific passage** inside known docs → `chunk`.
- "Find the **document** that matches this description" → `summary` (or `hybrid`).
- General natural-language questions → `hybrid` (recommended default).
- Max quality, have GPU/cloud → `hybrid` + `enable_reranker=true`.
- Pure CPU, privacy-first → `hybrid`, **reranker off**.
